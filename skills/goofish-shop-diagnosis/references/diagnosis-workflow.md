# 诊断工作流

## 触发信号 → 诊断路径

| 用户说 | 优先跑哪条 |
|---|---|
| "商品没曝光" / "发了没人看" | A. 单商品诊断 |
| "整个店铺都没流量" | B. 全账号诊断 |
| "之前好好的突然掉了" | C. 时序归因 |
| "同品别人卖得好我卖不好" | D. 对比诊断 |

## 路径 A · 单商品诊断

```
1. auth_status → 账号状态 OK？
2. item_view(item_id) → 拉商品详情
3. search_items(keyword=标题核心词) → 看排名
4. 按 traffic-drop-causes.md 逐项过
5. 输出归因 + 修复建议
```

**用时**：1-2 分钟
**调用次数**：2-3 次 MCP

## 路径 B · 全账号诊断

```
1. auth_status → 账号闲气值？（让用户告诉你，CLI 拿不到）
2. search_items(keyword="<卖家主要品类>", filter by seller=自己)
   → 看自家商品是否都能搜到
3. 如果多数搜不到 → 账号级降权，看 traffic-drop-causes 归因 6
4. 如果零散有搜不到的 → 看具体商品（回路径 A）
```

**用时**：3-5 分钟
**注意**：v0.2.4 没有"按 seller 筛"的直接 API，需要用"关键词 + 卖家 id"组合搜，有时不够精准。提醒用户这是经验诊断。

## 路径 C · 时序归因

用户描述了**时间点**变化（"上周还有流量，这周没了"）：

```
1. 用户报时间点
2. 问用户："那个时间前后你做了什么？" 
   - 改过价 / 标题 / 类目？
   - 发过多件新商品？
   - 和某个买家起争执？
   - 发消息时提过微信 / 线下？
3. 对应事件 → 对应归因
```

CLI 当前无时序数据存储，所以**靠问用户**而不是查接口。

**注意**：用户常记不清，不要强迫他们"精确到几号"。

## 路径 D · 对比诊断

用户提供**竞品链接**或**同类商品**：

```
1. item_view(竞品 id) → 拉竞品详情
2. item_view(自家 id) → 拉自家详情
3. 逐项对比：
   - 标题结构
   - 价格区间
   - 图片数量/首图
   - 描述长度 / 结构
   - 类目
4. 找出差距最大的 2-3 项 → 建议优先改
```

## 共用原则

### 输出格式固定

```
【诊断类型】路径 A/B/C/D
【核心发现】一句话结论（"降价改标是主因"）
【证据】
  - (item_view 里的具体字段)
  - (search_items 的排名结果)
【嫌疑归因 Top 3】
  1. ...
  2. ...
  3. ...
【建议操作】
  优先：...
  其次：...
  不要：...
```

### 诊断结束语模板

> "这是基于当前状态 + 社区经验规则的推断，不是 100% 确定。如果修复后 7-14 天还没恢复，可能是账号级问题，建议去闲鱼 App 的'违规记录'和'申诉'看下有没有具体通知。"

### 别忘记的

1. 有些问题不是"限流"而是**类目本身冷 / 挂价太高** → 诊断时别急着给"限流"帽子
2. 有些问题要**时间**而不是**动作** → 修完让用户静待
3. 有些问题 **CLI 根本看不到**（鱼塘权重、擦亮历史）→ 指路 App 后台

## 话术示例

### 开局
> "好的，我来帮你诊断一下。先让我看下这件商品的当前状态……"
> （跑 item_view + search_items）

### 中段
> "我看下来嫌疑最大的是 X，因为 Y。不过也有可能是 Z，我建议你先确认一下..."

### 收尾
> "按上面的优先级改，改完 2-3 天内不要再动它。如果还没起色，可能就得去 App 后台看有没有具体违规通知了。"

## 反模式

❌ **不要**给用户一个"权重公式"让他自己算——大部分用户不想看数学
❌ **不要**给 10+ 条建议——信息过载，挑 2-3 条
❌ **不要**让用户"等等看"就收尾——要给**明确下一步 + 时间点**
❌ **不要**跑 10 次 search_items 砸风控——买家视角搜 1-2 次够用
